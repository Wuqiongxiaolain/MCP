# 技术分析：vcpkg（包管理） vs 零第三方依赖

> latest update: v0.2.9-beta, 2026-07-17  
> 对照对象：graphmcp 当前依赖策略（手写 JSON / XML / Base64；无 vcpkg、无第三方库链接）

本文比较在 C++ 工程中引入 **vcpkg（或同类包管理器）拉取第三方库**，与坚持 **零第三方依赖（全部能力在树内实现或仅调用系统/外部进程）** 两条路线，并说明本项目坚持后者的具体考量。

### 决策摘要

| 项 | 内容 |
|----|------|
| **最终选择** | 零第三方 C/C++ 库链接；JSON/XML/Base64 树内实现 |
| **否决方案** | 默认以 vcpkg（或同类）拉取 nlohmann/json、sqlite3 等作为主依赖策略 |
| **约束条件** | 单文件交付；构建无包管理器前置；教学可审计；PNG/PDF 仅允许运行期外部进程委托 |
| **实现挂钩** | `src/json.hpp` 等自研头；`Makefile` 直接编译；版本见 `VERSION` |
| **可回退 / 重评条件** | 某能力被证明无法用合理子集替代（如可选 SQLite、完整 xlsx）时，可引入**单一、可静态链接、许可证清晰**的受控依赖，并更新构建/CD 文档 |

结构：问题界定 → 方案优势/代价 → 对比 → 选用与考量 → 结论。

---

## 1. 方案界定

| 维度 | vcpkg / 包管理驱动 | 零第三方依赖 |
|------|--------------------|--------------|
| 依赖获取 | `vcpkg install` / manifest 锁定版本 | 源码树内自包含实现 |
| 典型库例 | nlohmann/json、sqlite3、spdlog、asio 等 | `json.hpp`、迷你 XML、Base64 自研 |
| 构建入口 | CMake + `CMAKE_TOOLCHAIN_FILE` 或 overlay | Makefile / CMake 直接编译 `src/main.cpp` 一带入口 |
| 运行时 | 可能依赖动态库，或经静态链接并入产物 | 目标为单文件、尽量静态；无「再装一堆 dll」 |
| 升级路径 | 提升 port 版本并重建 | 自行维护与修补内置实现 |

「零依赖」在本项目中的精确含义是：**编译期不链接第三方 C/C++ 库**。PNG/PDF 等可通过本机已安装的浏览器无头模式等**外部程序**完成，失败时回退 SVG——这属于运行期可选委托，并不引入链接依赖，因而仍计入零第三方库策略。

---

## 2. vcpkg（包管理）方案：优势

### 2.1 工程生产率

- **复用经过验证的实现**：JSON 解析、HTTP、日志、压缩、加密等不必从零编写，能够显著缩短「可用功能」到达时间。
- **社区缺陷修复可跟随上游**：安全补丁与边界情况修复往往已由库维护者覆盖，项目可专注业务语义。
- **API 文档与示例丰富**：新人上手某一子系统时，可直接对照上游文档，而不必先通读自研头文件。

### 2.2 能力上限与生态

- **更容易接入重型能力**：完整 SQL 引擎、高性能 HTTP 服务器、成熟序列化（如 protobuf）等，用 port 引入通常比自研更现实。
- **跨平台差异由 port 吸收**：同一份 `vcpkg.json` 可在 Windows / Linux / macOS 上解析出对应 triplet，减少手写 `#ifdef` 的表面积。
- **版本可锁定**：manifest mode 将依赖版本写入仓库，有利于复现构建与安全审计（在锁文件被正确提交的前提下）。

### 2.3 团队协作

- **职责边界清晰**：业务代码与基础设施库分离；擅长业务的人不必同时维护 JSON 词法细节。
- **与现代 CMake 工作流对齐**：许多开源 C++ 项目默认假设使用者熟悉 vcpkg/Conan，贡献者心理成本更低。

---

## 3. vcpkg（包管理）方案：劣势

### 3.1 构建与环境成本

- **引导步骤变长**：贡献者须先安装 vcpkg、引导 bootstrap、配置 triplet 与工具链；「一条 `make all`」不再成立。
- **首次编译耗时显著增加**：拉取并编译依赖（尤其静态 triplet）可能消耗数十分钟与大量磁盘，CI 缓存策略必须同步设计。
- **环境漂移风险**：本机 vcpkg 版本、classic 与 manifest 混用、错误的 triplet，都会导致「别人能编、我不能」类问题，排障往往偏离业务本身。

### 3.2 交付与可移植性

- **动态链接时分发变复杂**：用户机器缺少对应运行时 DLL/so 时，程序无法启动；静态链接虽可缓解，但二进制体积与许可证合规审查压力随之上升。
- **许可证叠加**：每个引入的库都带来许可证义务（注意、归属、传染性条款等），课程项目亦须认真对待。
- **与「下载一个 exe 即用」叙事冲突**：一旦依赖链变长，Quick Start 中的零摩擦承诺更难维持。

### 3.3 控制权与课程约束

- **问题定位穿越多层抽象**：崩溃可能出在上游库内部；在教学场景下，学生更难完整理解调用栈与数据路径。
- **升级被动性**：上游 breaking change 会倒逼业务适配；不升级则累积 CVE 风险。
- **课题「可说明性」变弱**：评审时较难简洁地陈述「每一字节逻辑都在本仓库可审计」。

---

## 4. 零第三方依赖：优势

### 4.1 构建与交付一致性

- **克隆即可编**：仅需合规的 C++17 编译器（及 Make/CMake）；Windows MinGW、Linux、macOS CI 矩阵可共享同一心智模型。
- **单文件可移植叙事成立**：静态或近静态链接后，用户无需安装包管理器或额外运行时库即可运行 MCP 服务器。
- **CI 路径短且稳定**：构建步骤不依赖外部 port 仓库可用性，减少「因 vcpkg registry 抖动导致 CI 红」的外部因素。

### 4.2 架构叙事统一

- **与单 exe、JSON 文件存储同一哲学**：少运维、可审计、本机优先；依赖策略不会在中途把项目拖向「服务端中间件栈」。
- **协议与格式栈可端到端掌控**：MCP JSON-RPC、图/表模型序列化、SpreadsheetML 子集等均在树内实现，行为变更可精确提交与回滚。
- **教学与答辩友好**：能够清晰演示「解析器如何工作、存储目录如何演进」，而不是「我们调用了某某库的某某 API」。

### 4.3 风险面收敛

- **供应链攻击面更小**：无需信任大量上游 port 与其传递依赖。
- **许可证简单**：以本仓库代码为主，合规讨论范围可控。
- **缺陷归属明确**：解析错误、序列化不一致等问题，修复路径直接落在本项目源文件。

---

## 5. 零第三方依赖：劣势

### 5.1 实现与维护成本

- **重复造轮子的时间成本真实存在**：自研 JSON/XML 须覆盖足够测试；边界情况（BOM、转义、深层嵌套、大文件）要靠本项目持续补齐。
- **功能上限受自制组件约束**：例如完整 `.xlsx`、高并发网络服务、复杂全文检索等，在零依赖前提下实现成本陡增，往往只能做成子集或明确不做。
- **性能调优起点较低**：成熟库常带有多年优化；自研实现须靠基准测试（本项目已有 18 指标微基准）主动守住回归。

### 5.2 人才与生态

- **贡献者门槛形态变化**：不熟悉本仓库 JSON API 的人，不能直接套用 nlohmann 经验；需阅读项目内头文件与注释。
- **安全响应全靠自身**：若自研解析器出现漏洞，没有上游发版可跟，必须自行修补并发布新二进制。

### 5.3 与「适度引入」的张力

- 项目后续若评估 **SQLite 可选后端**（见 [JSON vs SQLite](ANALYSIS_JSON_VS_SQLITE.md)），将不可避免地重新打开「是否允许一个受控依赖」的讨论；零依赖是当前基线，而非永远不可调整的教条。

## 6. 对照表（决策用）

| 评价项 | vcpkg / 第三方库 | 零第三方依赖 | 对本项目的权重 |
|--------|------------------|--------------|----------------|
| 首次构建摩擦 | 高 | 低 | **极高** |
| 单文件分发可信度 | 视链接方式而定 | 高 | **极高** |
| 开发速度（通用基建） | 高 | 中低 | 中 |
| 行为可审计性 / 教学演示 | 中 | 高 | **高** |
| 重型能力天花板 | 高 | 需取舍 | 中（当前需求可覆盖） |
| CI 外部依赖稳定性 | 中（受 registry/缓存影响） | 高 | 高 |
| 许可证与供应链复杂度 | 高 | 低 | 高 |
| 与 MCP 本地工具定位一致性 | 弱相关 | 强相关 | 高 |

## 7. 本项目的选用与考量

**选用方案：零第三方 C/C++ 库依赖；JSON / XML / Base64 均在仓库内实现。**

主要考量如下：

1. **课题与产品同时强调「可下载即用」**  
   Quick Start 明确写明：无需克隆整仓，一个静态编译的可执行文件即是完整 MCP 服务。引入 vcpkg 会把「能编出来」与「能跑起来」拆成两段故事，削弱该承诺。

2. **核心工作量应落在领域模型，而非依赖治理**  
   真正的价值在于统一 Graph/Table、多格式往返、版本与游标、MCP 工具面与布局。把日历时间花在 port 冲突、triplet 与 CI 缓存上，投入产出比对课程主线不利。

3. **所需第三方能力可用「自制子集 + 外部进程」覆盖**  
   - 序列化：自研 JSON 足够支撑模型与 MCP 报文；  
   - 表交换：SpreadsheetML 2003 子集即可被 Excel 打开，无需完整 OOXML 库；  
   - 栅格化：委托本机浏览器无头能力，失败回退 SVG。  
   因此「缺库」并未在首期构成硬阻塞。

4. **质量靠测试与基准托住，而非靠「用了名库」托住**  
   单元测试、冒烟、MCP 协议测试与微基准回归，用来约束自研组件的正确性与性能；这与零依赖策略是配套设计，而不是互相替代。

5. **与单 exe、JSON 存储形成闭环叙事**  
   评审与文档中可以一以贯之地说明：进程形态简单、依赖形态简单、持久化形态简单。三者互相加强，避免「本地工具却挂着一长串动态库与数据库守护进程」的错位。

**明确未默认采用 vcpkg 的原因（简述）**：通用库带来的速度优势，不足以抵消本项目对构建摩擦、单文件交付与教学可审计性的硬约束；在需求仍可用自制子集满足时，引入包管理器属于过早优化工程复杂度。

**可演进原则**：若未来某项能力（例如可选 SQLite 后端、完整 xlsx）被证明**不可**用合理子集替代，应优先评估「单一、许可证清晰、可静态链接」的受控依赖，并同步更新构建文档与 CD 矩阵——而不是一次性放开无约束的依赖增长。

---

## 8. 结论

vcpkg 适合需要快速堆叠通用基础设施、团队已熟悉 C++ 包管理、且接受较重环境引导的产品工程。graphmcp 当前阶段的目标是 **可移植的本地 MCP/CLI 工具**，因此更应把复杂度留在领域代码内，并把依赖面压到零。零依赖不是否认第三方库的价值，而是在明确的交付与教学约束下，**主动选择把控制权与可说明性放在第一位**。
