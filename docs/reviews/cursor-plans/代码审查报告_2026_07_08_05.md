# 代码审查报告_2026_07_08：Rough 渲染审计与清理

## 结论（先给判断）

当前 PNG 走的是 **C++ 静态“抖动折线逼近”**，不是 Excalidraw 网页端的 **rough.js 曲线生成**。因此：
- 效果注定弱（样例里几乎全是 `roughness: 1`，抖动幅度约 `strokeWidth * 0.14` ≈ 0.3px 量级）；
- 与网页预览 **不可能视觉一致**；
- 若要对齐网页，需可靠跑通真实 rough.js（或完整移植其几何），复杂度明显高于本项目当前架构收益。

**默认决策：删除静态 rough 叠加功能**（含 HTML 演示路径的 rough 依赖测试调整），PNG/PDF 继续用精确 SVG → 栅格化。

---

## 1. 摘出：所有控制 rough 效果的函数/入口

均集中在 [`src/exporters.hpp`](src/exporters.hpp)：

| 符号 | 作用 |
|---|---|
| `vendoredRoughJs()` | 读入 `third_party/excalidraw-assets/rough.js`（**仅**给 `toExcalidrawRoughHtml`；**PNG 路径未用**） |
| `roughSeedState` | 由 `seed`/`id` 生成 PRNG 状态 |
| `roughRand01` | LCG 随机数 |
| `roughRoundedRectPoints` | rough 层圆角矩形采样点 |
| `roughElementPressures` | 读 `pressures` |
| `roughElementRoughness` | 读 `roughness`（0–8） |
| `roughElementThinning` | 读 `strokeOptions.thinning` |
| `toSVGExcalidraw(..., addRoughOverlay)` | 精确层 + 可选 `rough-overlay`；核心逻辑为内部 lambda **`emitRoughVariableStroke`** |
| `toExcalidrawRoughHtml` | 内联 rough.js 的 HTML（**导出 PNG/PDF 已不走这条路**） |
| `exportGraph` 的 `png/pdf` 分支 | 白板时强制 `toSVGExcalidraw(g, true)` |

数据流（与网页不一致的关键点）：

```mermaid
flowchart LR
  input[Excalidraw JSON]
  graph[Graph.elements]
  svgExact[toSVGExcalidraw exact layer]
  svgRough[emitRoughVariableStroke overlay]
  raster[rasterize SVG to PNG]
  web[Browser Excalidraw]
  roughjs[rough.js curve ops]

  input --> graph
  graph --> svgExact
  graph --> svgRough
  svgExact --> raster
  svgRough --> raster
  input --> web
  web --> roughjs
```

测试侧相关断言在 [`tests/test_main.cpp`](tests/test_main.cpp)：`toExcalidrawRoughHtml` 字符串检查 + `id="rough-overlay"` / roughness 输出差异。

## 2. 分析：rough“几乎无效果”且与网页不一致的原因

### A. 实现算法就不是 rough.js
- Excalidraw 网页：`rough.svg(svg)` → `rectangle/ellipse/polygon/linearPath`，含 bowing、多笔触随机曲线、填充纹理、端点与拐角语义。
- 本仓库 PNG：对轮廓做 **少量采样点随机偏移 + 分段 `<line>` 变线宽**。
- 结果：顶多像“折线微抖”，不是手绘笔触。

### B. 参数量级天然偏弱（以样例为准）
样例 [`examples/example_input/whiteboard_freedraw.excalidraw`](examples/example_input/whiteboard_freedraw.excalidraw) 元素基本是 `"roughness": 1`。当前公式形如：

```text
roughAmp = max(0.12, strokeWidth * (0.05 + roughness * 0.09))
# roughness=1, sw=2 → amp ≈ 0.28px
```

在 `force-device-scale-factor=2` 截图后再看，抖动常被稀释成肉眼几乎不可辨。

### C. 分段直线无法表达“连续 pen 感”
矩形/椭圆边被拆成多段 `<line>`，每段独立 `stroke-width`；缺少：
- rough.js 的多 stroke path、bowing、角点不规则；
- 端点圆帽/重叠策略与 Excalidraw 一致；
- fillStyle（hachure 等）。

### D. 导出路径与“网页预览”不是同一链路
- PNG：`exportGraph` → `toSVGExcalidraw(g,true)` → `rasterize`（浏览器只截 **静态 SVG**）。
- `toExcalidrawRoughHtml`（真实 rough.js）已因白屏问题未用于 PNG；即便恢复，也仍要解决 font/image ready、脚本完成等待等问题。

### E. 精度代价高
要对齐网页，现实路径只有：
1. **CDP/Playwright 等待 `__graphmcp_render_ok` + 真实 rough.js**（再处理 image/files）；或
2. **完整移植 rough.js/excalidraw 几何到 C++**。

二者相对本仓库规模，维护成本和回归面都过大。

## 3. 计划执行：删除近似 rough（默认）

确认后改动范围：

1. **[`src/exporters.hpp`](src/exporters.hpp)**
   - `exportGraph(png/pdf)`：白板改回 `toSVG(g)` / `toSVGExcalidraw(g, false)`，不再强制 `addRoughOverlay=true`。
   - 删除或内联移除：`roughSeedState`、`roughRand01`、`roughRoundedRectPoints`、`roughElementPressures`、`roughElementRoughness`、`roughElementThinning`、`emitRoughVariableStroke` 整块、`toSVGExcalidraw` 的 `addRoughOverlay` 参数与分支。
   - 删除 `toExcalidrawRoughHtml` 与 `vendoredRoughJs` 对导出主路径的耦合（若无其他调用则一并删除函数；`third_party/excalidraw-assets/rough.js` 可保留为未用资源或后续再清，本次以代码引用归零为主）。
   - 形状精确层恢复始终画 stroke（去掉 `addRoughOverlay ? "none"` 逻辑）。

2. **[`tests/test_main.cpp`](tests/test_main.cpp)**
   - 去掉 `rough-overlay` / roughness 差异断言。
   - 调整/删除仅针对 `toExcalidrawRoughHtml` 的用例（若函数删除）。

3. **验证**
   - `cmake --build` + `ctest`
   - 重新导出 whiteboard PNG，确认无白屏、图面为精确几何（可接受与网页笔触风格差异，但无错误叠加阴影）。
