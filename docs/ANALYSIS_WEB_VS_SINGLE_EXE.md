# 技术分析：Web 前后端分离 vs 单可执行文件

> latest update: v0.2.9-beta, 2026-07-17  
> 对照对象：graphmcp 当前交付形态（C++17 单可执行文件 + CLI / MCP stdio）

本文从部署形态、交互模型、运维成本、安全边界与课题约束五个维度，系统比较 **Web 前后端分离架构** 与 **单可执行文件（single binary）** 两种方案，并说明本项目最终采用后者的具体考量。

### 决策摘要

| 项 | 内容 |
|----|------|
| **最终选择** | 单可执行文件（CLI + `graphmcp serve` MCP） |
| **否决方案** | Web 前后端分离（浏览器 SPA + 独立后端）作为主交付形态 |
| **约束条件** | 课程周期；MCP stdio 本地子进程模型；可下载即用、少运维 |
| **实现挂钩** | `src/main.cpp`、`src/mcp.hpp`；制品为 `bin/graphmcp`；版本见根目录 `VERSION` |
| **可回退 / 重评条件** | 出现明确的多租户在线协作或「必须自建一等画布后端」需求时，单独 ADR 重评；轻量预览可先评估本地静态页 + 监听 `latest.json` |

结构：问题界定 → 方案优势/代价 → 对比 → 选用与考量 → 结论。

---

## 1. 方案界定

| 维度 | Web 前后端分离 | 单可执行文件 |
|------|----------------|--------------|
| 典型形态 | 浏览器 SPA/SSR + 独立后端服务（HTTP/WebSocket） | 一份静态链接的 `graphmcp` / `graphmcp.exe` |
| 用户入口 | URL、登录态、页面路由 | CLI 子命令；MCP 客户端经 stdio 拉起进程 |
| 进程模型 | 常驻服务（多实例/反向代理可选） | 按需启动；MCP 场景下由宿主托管生命周期 |
| 状态与会话 | 服务端会话或 JWT；跨请求共享连接池 | 进程内状态；持久化落在本地 `GRAPHMCP_STORE` |
| 交付物 | 前端构建产物 + 后端二进制/容器 + 配置与网关 | 单一可执行文件（可选文档与配置模板） |

对本项目而言，「单可执行文件」并不排斥调起外部编辑器或浏览器查看 SVG：可视化仍可委托 Draw.io / Excalidraw / 系统浏览器，但**业务逻辑与 MCP 协议实现始终驻留在本地进程内**，而不必另行部署一套 Web API。

---

## 2. Web 前后端分离：优势

### 2.1 交互与可达性

- **跨终端访问**：用户仅需浏览器即可操作，不必预先安装与本机工具链匹配的二进制，从而显著降低首次试用门槛。
- **富交互界面**：画布缩放、拖拽、实时预览、协作光标等能力可直接复用成熟的 Web 图形栈，交互反馈能够更细腻地呈现。
- **多用户并发**：通过鉴权、租户隔离与共享后端，同一服务实例可同时服务多名用户，天然贴合「团队共创图库」一类产品形态。

### 2.2 演进与分工

- **前后端独立迭代**：UI 与协议层可按不同节奏发版；前端热更新不必强制用户重装后端。
- **技术选型更灵活**：前端可采用 TypeScript + 图形库；后端可选用任意语言与 ORM，团队可按专长拆分职责。
- **水平扩展路径清晰**：无状态 API + 外部存储时，可通过增加副本与负载均衡线性扩展吞吐。

### 2.3 运维与观测

- **集中式日志与指标**：请求链路、错误率、延迟百分位可统一汇入 APM；问题定位通常不依赖用户本机环境。
- **灰度与回滚**：借助网关与版本化部署，可逐步放量并在异常时迅速切回上一版本。

---

## 3. Web 前后端分离：劣势

### 3.1 复杂度与成本

- **组件显著增多**：至少需要前端构建流水线、后端进程、反向代理、HTTPS 证书、CORS 与鉴权策略；本地开发亦须同时拉起多服务。
- **环境与配置漂移**：开发、预发、生产之间的环境变量、数据库连接与静态资源路径更容易出现不一致，排障路径变长。
- **持续运维开销**：服务需保持可用；磁盘、内存、补丁与依赖升级形成长期负担，对课程实践周期而言往往过重。

### 3.2 与 MCP / AI 宿主的契合度

- **MCP 主流传输为 stdio**：Claude Code / Claude Desktop 等客户端默认以子进程 + 标准输入输出对接工具服务器；额外引入 HTTP 网关会强制增加一层适配与鉴权，并不能直接简化接入。
- **延迟与往返**：浏览器 → 网关 → API → 存储的多跳路径，对「Agent 高频、小粒度改图」并不友好；本地进程内调用通常能更稳定地控制尾延迟。
- **离线与内网场景受限**：无公网或严格隔离的环境中，Web 服务要么无法触达，要么必须额外部署私有化套件。

### 3.3 安全与数据边界

- **攻击面扩大**：开放端口、XSS、CSRF、注入与错误的 CORS 配置都会进入威胁模型；单机 CLI/MCP 进程的默认暴露面明显更小。
- **数据驻留策略变复杂**：图与表若上云，需明确备份、删除权与合规要求；本地文件存储则把边界收敛到用户工作区。

## 4. 单可执行文件：优势

### 4.1 交付与可移植性

- **一份产物即可运行**：用户下载 `graphmcp.exe`（或 Linux/macOS 对应二进制）后即可作为 CLI 或 MCP 服务器使用，无需先配置 Node/Python 运行时或数据库守护进程。
- **跨平台构建矩阵可控**：在 GitHub Actions 上为 Windows / Linux / macOS 分别产出静态或近静态链接制品，发布路径与课题 CD 要求高度对齐。
- **与「零第三方依赖」策略互洽**：无动态库清单时，单文件交付的承诺更容易真正兑现（详见 [vcpkg vs 零依赖](ANALYSIS_VCPKG_VS_ZERO_DEP.md)）。

### 4.2 与产品定位匹配

- **MCP 工具服务器本就是本地进程**：`graphmcp serve` 经 JSON-RPC over stdio 与 AI 客户端对话；单 exe 是该模型的自然载体，而不是「再包一层 Web」。
- **CLI 与 MCP 共用同一套逻辑**：解析、布局、存储、导出均编译进同一二进制，避免前后端两套模型漂移。
- **外部可视化按需委托**：需要人工精修时，通过 `graph_open` / `edit` 调起已安装的 Draw.io / Excalidraw / 浏览器；本工具专注结构化变换与版本，而不重复实现完整在线画布。

### 4.3 开发与验证效率

- **构建—测试—冒烟链路短**：`make all && make test && make smoke` 即可在本地闭环；CI 不必同时维护前端 e2e 与后端集成两套主路径。
- **问题复现成本低**：用户提供命令行参数、环境变量 `GRAPHMCP_STORE` 与一份图库目录，即可较为忠实地复现缺陷。

---

## 5. 单可执行文件：劣势

### 5.1 交互能力边界

- **缺少一等 Web UI**：实时画布预览、多用户同屏编辑等能力需另开路线（项目已将「实时画布预览」列为可选后续目标，设想为本地 HTML 轮询 `latest.json`，仍非完整前后端分离）。
- **分发依赖各平台产物**：虽是单文件，仍须为三平台分别构建与签名（若需要）；用户误下错架构包时体验会短暂受挫。

### 5.2 扩展与协作

- **多用户共享图库需额外设计**：默认模型是「本机目录 = 图库」；若要做成团队中心仓库，仍须叠加同步、锁或远端存储，此时 Web/服务端方案的优势会重新显现。
- **热更新不便**：修复缺陷通常意味着用户更换二进制；无法像 Web 前端那样静默推送静态资源。

### 5.3 进程模型约束

- **长连接与推送能力弱于常驻 WebSocket 服务**：MCP stdio 会话随宿主生命周期结束；不适合作为面向公网的常开协作后端。
- **资源隔离依赖操作系统**：恶意或异常大的输入会直接影响本机进程；Web 方案可通过沙箱与配额更细致地限制租户。

## 6. 对照表（决策用）

| 评价项 | Web 前后端分离 | 单可执行文件 | 对本项目的权重 |
|--------|----------------|--------------|----------------|
| 首次安装成本 | 低（浏览器） / 运维侧高 | 下载一个文件即可 | 高 |
| MCP / AI 宿主接入 | 需额外 HTTP 适配 | 原生 stdio 契合 | **极高** |
| 课程周期内可交付性 | 容易膨胀 | 易收敛到可演示主链 | **极高** |
| 富交互画布 | 强 | 弱（可外挂编辑器） | 中（非首期硬需求） |
| 多用户 SaaS | 强 | 弱 | 低（当前非目标） |
| 攻击面与合规复杂度 | 高 | 相对低 | 高 |
| 与零依赖 / JSON 文件存储一致性 | 弱耦合 | 强耦合、叙事统一 | 高 |

## 7. 本项目的选用与考量

**选用方案：单可执行文件（CLI + `graphmcp serve` MCP 服务器）。**

主要考量如下：

1. **协议契合优先于界面形态**  
   课题与产品定位的核心是「图形设计与绘图 **MCP 工具**」。MCP 客户端以本地子进程方式调用工具服务器；将能力收敛进单二进制，能够最直接地满足该调用模型，而不必先建设一套与 stdio 平行的 HTTP API。

2. **交付承诺可验证**  
   文档与 Quick Start 明确宣称：无需克隆整仓即可下载 exe 使用。单文件交付使该承诺可被用户立即验证；Web 方案则会把「能跑起来」推迟到服务部署完成之后。

3. **复杂度预算服从课程实践周期**  
   项目需在有限周期内打通「多格式解析 → 统一模型 → 布局导出 → 版本存储 → MCP/CLI」。前后端分离会显著占用本应用于图模型、Mermaid 深解析与表协作的时间预算。

4. **可视化并不等同于自建 Web 后端**  
   通过调起外部编辑器与导出 SVG/PNG/PDF，已经覆盖「给人看、给人改」的主路径；自建前后端画布属于增强项，而非替代 MCP 工具身份的前提。

5. **与另外两项架构决策一致**  
   零第三方依赖与 JSON 文件持久化都强调「本机、可审计、少运维」。单 exe 是这一组合在进程与分发层上的自然落点；若改为 Web 服务，存储与依赖策略通常也会被迫向服务端栈倾斜。

**明确未选择 Web 前后端分离的原因（简述）**：多用户 SaaS、浏览器内一等画布、集中式灰度发布等收益，当前均不构成项目的主验收路径；其引入的运维、鉴权与适配成本，会不成比例地挤占核心链路的完成度。

**可演进空间**：若未来需要「实时画布预览」，可优先评估**本地静态页 + 轮询/监听 `latest.json`** 的轻量方案，在不推翻单 exe 主形态的前提下补齐观感；仅当出现明确的多租户在线协作需求时，再单独论证是否引入完整前后端分离。

---

## 8. 结论

对 graphmcp 这类以 **MCP/CLI 驱动结构化图与表变换** 的工具而言，单可执行文件在接入路径、交付可验证性与复杂度控制上，明显更匹配当前目标；Web 前后端分离更适合以浏览器协作为中心、需要长期在线运维的产品。二者并非绝对优劣，而是**问题域不同**：本项目主动选择把问题域钉在「本地工具服务器」，因此坚定地采用单 exe，并将富 UI 留给外部编辑器或后续可选增强。
