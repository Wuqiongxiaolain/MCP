# 技术分析：JSON 文件持久化 vs SQLite

> latest update: v0.2.9-beta, 2026-07-17  
> 对照对象：graphmcp 当前存储实现（`GRAPHMCP_STORE` 下图/表目录并列的 JSON 快照体系）

课题需求允许 **JSON 或 SQLite**。本文从数据模型契合度、一致性、查询能力、运维与零依赖约束等角度，系统比较两种持久化方案，并说明本项目选用 JSON 文件系统的具体考量。

### 决策摘要

| 项 | 内容 |
|----|------|
| **最终选择** | 文件系统 JSON 快照（`latest` / `versions/vN` / draft·stage·HEAD） |
| **否决方案** | 首期以 SQLite 为默认唯一存储引擎 |
| **约束条件** | 与零依赖、单 exe 一致；主路径为按 id 整图/整表加载；MCP/Agent 需可读可 diff |
| **实现挂钩** | `src/storage.hpp`、`src/table_storage.hpp`；环境变量 `GRAPHMCP_STORE`；版本见 `VERSION` |
| **可回退 / 重评条件** | 基准/压力测试证明列表检索或写放大成为主矛盾时，评估可选 SQLite 或混合索引，并提供迁移策略 |

结构：问题界定 → 方案优势/代价 → 对比 → 选用与考量 → 结论。

---

## 1. 方案界定

| 维度 | JSON 文件持久化 | SQLite |
|------|-----------------|--------|
| 物理形态 | 目录树 + `index.json` / `latest.json` / `versions/vN.json` 等 | 单一（或少数）`.db` 文件内的表与索引 |
| 访问方式 | 进程内读写文件；可用普通编辑器打开 | SQL 语句；需库或 CLI（`sqlite3`）查看 |
| 事务 | 靠写文件顺序、临时文件与替换等约定模拟 | ACID 事务由引擎提供 |
| 查询 | 按路径/id 定位；列表依赖索引文件或目录遍历 | 声明式查询、索引、聚合 |
| 与依赖策略关系 | 纯标准库文件 IO + 自研 JSON | 通常需链接 sqlite3（或 amalgamation 源码纳入工程） |

本项目图侧另有 `draft.json` / `stage.json` / `HEAD` 等版本工作流文件；表侧目录并列、语义相近但无 Draft/Stage。下文的「JSON 方案」均指这一整套**文件系统快照模型**，而非「把整个库塞进一个巨大 JSON 数组」。

---

## 2. JSON 文件持久化：优势

### 2.1 可观测性与可调试性

- **内容可直接审阅**：用任意文本编辑器或 `git diff` 即可查看某版本图/表的完整快照，排障时不必先构造 SQL 或启动数据库工具。
- **与 Git 工作流天然亲近**：版本目录中的 JSON 可被版本控制系统跟踪（若用户选择把 store 纳入仓库）；冲突与变更以文本形式呈现，认知负担更低。
- **教学演示直观**：答辩或文档中可以直接展示目录结构与字段，帮助读者理解 Draft→Stage→Commit 的状态迁移。

### 2.2 运维与部署简单

- **零配置启动**：指定 `GRAPHMCP_STORE`（或使用默认 `./graph-store`）即可工作，无需初始化 schema、迁移工具或守护进程。
- **备份即复制目录**：拷贝 store 根目录等于备份；回滚也可通过替换文件或调用已有 `rollback`/`checkout` 语义完成。
- **损坏面局部化**：单个 `vN.json` 损坏通常只影响该版本；索引损坏时仍可能从 versions 目录尝试恢复（视实现与备份策略而定）。

### 2.3 与领域模型同构

- **文档型数据贴合图/表快照**：一次 `save`/`commit` 本质是写入一整个 Graph/Table 文档；JSON 快照与内存模型几乎一一对应，避免范式拆分带来的组装成本。
- **格式演进路径清晰**：增加字段（如颜色、waypoints、layers）时，多为向前兼容的键扩展；迁移往往体现为「读入缺省 + 写回补全」，而不必立刻编写多步 SQL migration。
- **与零依赖、单 exe 策略一致**：不引入数据库引擎，巩固「下载即用」的交付承诺。

---

## 3. JSON 文件持久化：劣势

### 3.1 查询与规模

- **缺乏声明式查询**：按标签模糊搜索、跨图统计、复杂过滤等，需在应用层扫描索引或加载文档，大规模时 CPU 与 IO 开销上升更明显。
- **写放大与全量快照**：每次提交保存整图/整表 JSON；节点微调也会复制大块未变更内容，磁盘与耗时随对象增大而增长（项目已在 MCP 路径上针对性削减写放大，但模型本质仍是文档快照）。
- **并发控制薄弱**：多进程同时写同一 store 时，文件替换与索引更新的竞态需自行严谨处理；SQLite 在同一文件上的锁与事务语义更现成。

### 3.2 一致性与工具链

- **跨文件原子性需自行设计**：`HEAD`、`latest.json`、`versions/vN.json`、`index.json` 之间的更新顺序若被中断，可能出现短暂不一致；应用必须定义清晰的写入协议。
- **二次分析工具少**：生态中针对「通用业务库」的 BI/报表工具默认面向 SQL；对目录型 JSON 库，分析脚本要额外编写。

### 3.3 长期演进压力

- 当图/表数量与单对象体积进入「系统性压力测试」区间后，列表、历史浏览与检索的体验可能先于功能正确性成为瓶颈；项目已将「大图/大表压力测试」与「SQLite 可选后端」列为后续评估项。

---

## 4. SQLite：优势

### 4.1 引擎级能力

- **ACID 事务**：多表更新可在同一事务中提交或回滚，崩溃一致性由引擎与日志机制支撑，应用层协议更轻。
- **索引与查询**：对 id、名称、类型、时间戳等建立索引后，列表与过滤可保持较稳定的延迟曲线。
- **单文件备份**：一个 `.db`（外加足够的 `-wal`/`-shm` 认知）即可带走全部数据，分发与拷贝仍然简单。

### 4.2 生态与扩展

- **工具链成熟**：`sqlite3` CLI、GUI 浏览器、ORMs、迁移框架齐全，便于运维与只读分析。
- **并发读更友好**：在常见 journal/WAL 模式下，多读者场景通常优于「多进程同时解析大 JSON」的朴素实现。
- **易于做关系型投影**：若未来大量「图↔表」分析希望落在 SQL，可把边列表、节点属性规范化进表，分析语句会更短。

### 4.3 规模化潜力

- 在对象数量上升、需要局部更新而非整文档重写时，行级更新往往比重写整份 JSON 更节省 IO——前提是数据模型已经合理地拆到表结构中。

---

## 5. SQLite：劣势

### 5.1 依赖与复杂度

- **引入库或 amalgamation**：即使用官方 amalgamation 把源码编进工程，也仍然是「第三方代码进入构建」；与当前零依赖基线存在张力（见 [vcpkg vs 零依赖](ANALYSIS_VCPKG_VS_ZERO_DEP.md)）。
- **schema 与迁移**：字段演进需要版本化 migration；比「JSON 多一个键」更重，做错迁移的代价更高。
- **文档型数据需拆解**：Graph 含嵌套节点、边、白板 elements、多层 pages 等，完整范式化会导致大量表与 JOIN；若改为 JSON 列存进 SQLite，则又回到「库里套文档」，收益打折。

### 5.2 可观测性下降

- **内容不再默认人类可读**：二进制库文件无法直接 `cat`；调试须借助工具，AI Agent 与人类在排障时的即时可见性变差。
- **与 Git diff 不友好**：除非额外导出文本快照，否则代码评审式的「看一眼变更了什么」变难。

### 5.3 产品契合度问题

- **对本项目的主路径并非刚需**：当前工具按 id 打开图/表、读写当前版本、追加快照——访问模式以主键点查与整文档读写为主，SQL 的表达力优势大部分时间用不上。
- **运维心智增加**：锁、WAL 检查点、损坏修复（`integrity_check`）等知识进入维护清单；对「本地 MCP 小工具」略显过重。

## 6. 对照表（决策用）

| 评价项 | JSON 文件持久化 | SQLite | 对本项目的权重 |
|--------|-----------------|--------|----------------|
| 零配置 / 零依赖契合 | 高 | 中低 | **极高** |
| 人类可读与 Git 友好 | 高 | 低 | **高** |
| 整文档快照与版本模型契合 | 高 | 中（需设计） | **极高** |
| 复杂查询与局部更新 | 低 | 高 | 中（当前偏低） |
| 崩溃一致性 / 多进程写 | 中（靠约定） | 高 | 中 |
| 教学与答辩可说明性 | 高 | 中 | 高 |
| 超大规模潜力 | 中低 | 高 | 低～中（后续项） |
| 实现与迁移成本（当前周期） | 低 | 中高 | **极高** |

## 7. 本项目的选用与考量

**选用方案：基于文件系统的 JSON 持久化（图与表目录并列；版本快照 + 索引 + HEAD）。**

主要考量如下：

1. **需求允许二选一，决策应服从主访问模式**  
   课题写明「JSON 或 SQLite」。本工具的主路径是按 id 加载/保存整份 Graph 或 Table，并保留不可变版本快照——这与「一版本一 JSON 文档」高度同构，而不是与高度规范化的关系模型同构。

2. **可调试性直接服务 MCP / Agent 场景**  
   Agent 与开发者经常需要确认「当前 draft 究竟写成了什么」。文本快照使人工抽查、脚本对比与失败复现都更直接；若默认藏进 `.db`，排障步骤会系统性变长。

3. **巩固零依赖与单文件交付**  
   JSON 方案只依赖标准文件 IO 与自研解析，不必链接 sqlite3，从而与 [零依赖策略](ANALYSIS_VCPKG_VS_ZERO_DEP.md)、[单 exe 交付](ANALYSIS_WEB_VS_SINGLE_EXE.md) 保持同一约束集。

4. **版本语义可以用目录协议清晰表达**  
   `draft` / `stage` / `versions/vN` / `HEAD` / `latest` 的分工用文件角色即可讲清；等价语义若映射到 SQLite，仍需精心设计表结构与事务边界，首期收益有限而设计成本更高。

5. **已知边界被显式记录，而非忽视**  
   项目全景已将「大图/大表系统性压力测试」与「SQLite 可选后端」列为后续目标：即承认 JSON 在检索与规模上的天花板，但把切换时机放在**有证据表明瓶颈真实存在**之后，而不是在缺乏负载数据时预先引入引擎。

**明确未首期采用 SQLite 的原因（简述）**：ACID 与 SQL 查询的收益，对当前「文档快照 + id 点查」主路径并不关键；其依赖、schema 与可读性代价，反而冲击零依赖交付与教学可说明性。需求文档中的「或」字，本项目解释为**允许选择 JSON 并把它做完整**，而非必须同时实现两种后端。

**可演进方向**：若基准与压力测试表明列表/检索/写放大成为主矛盾，可评估：

- **保留 JSON 为默认**，增加可选 SQLite 后端（配置开关）；或  
- **混合**：索引与元数据进 SQLite，大文档仍外置 JSON/BLOB，以兼顾查询与可读性。  

任何切换都应同步给出迁移工具与兼容策略，避免打断已有 `GRAPHMCP_STORE` 用户。

---

## 8. 结论

JSON 文件持久化以可读性、零配置和与文档型版本模型的同构性取胜；SQLite 以事务、索引与规模化查询取胜。graphmcp 在课程周期与 MCP 本地工具定位下，**合理地优先选择了前者**，并把后者保留为有数据支撑后再评估的可选后端。这不是否定 SQLite，而是坚持：**存储引擎应当匹配当前访问模式，而不是匹配抽象的「数据库更专业」直觉。**
